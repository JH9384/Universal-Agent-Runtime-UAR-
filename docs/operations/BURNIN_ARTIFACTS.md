# Burn-In Artifact Model

Burn-in artifacts provide operational evidence for runtime survivability.

## Artifact Categories

### Runtime Health
- pressure traces
- oscillation traces
- starvation incidents
- operating-mode transitions

### Replay
- divergence reports
- replay confidence summaries
- reconstruction artifacts
- continuity validation

### Topology
- partition events
- fanout spikes
- congestion traces
- isolation incidents

### Long-Horizon Stability
- soak-run histories
- memory drift traces
- reconnect pressure histories
- convergence timelines

## Operational Purpose

Artifacts are intended for:

- runtime certification
- operational debugging
- convergence analysis
- replay archaeology
- survivability validation

## Principle

Artifacts should remain:

- deterministic where possible
- append-oriented
- replay-safe
- pressure-bounded
- operator-readable
